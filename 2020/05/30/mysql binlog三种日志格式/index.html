MySQL binlog三种日志格式

一.MySQL binlog日志格式

  1. mySQL binlog日志总共分为三种：ROW，Statement，MiXED

  2. 查询自己mysql使用的日志格式：

    show variables like ‘%binlog_format%’

  3. 切换模式

    set binlog_format = statement

    ##修改/etc/my.cnf

    [mysqld]

    binlog_format = ROW //定义mysql默认日志格式

二. 三种格式详细分析

1. ROW

​ row格式文件体积大，记录数据行的变化，不会记录执行的sql语句和上下文信息，仅仅保留哪条记录经过修改。该格式也是mysql默认的日志格式，

优点：能够清晰的记录每一行数据的修改动态细节，不会出现statement格式中function、存储过程或trigger触发后无法正确复制的情况。

缺点： 磁盘io损耗较大，同一个事务若进行多条数据的修改，则会产生大量的日志内容，若主从数据库存在硬件差异，进行大量数据同步时即会产生主从延迟；因为在主从同步的场景中，row格式的binlog event在执行时会翻译成类似于sql语句的方式逐条执行，若没有建立索引或分区度太低，执行效率会非常缓慢。（通常解决方法见下文）

2. Statement

​ statement将每一条修改的sql语句记录至binlog中

优点：节约了磁盘IO的损耗，性能相比row格式较高（主要指update、alter和整表删除的场景，row格式会产生大量的日志内容）

缺点： 只记录执行sql语句，一些函数语句无法正确复制到从库，容易产生数据丢失；故还需要记录每条语句执行时的一些相关信息（function，存储过程或trigger）才能正确同步数据。

3. MiXED

​ 以上两种的混用，一般语句修改采用statement格式保存，涉及一些函数内容statement无法正确主从同步则采用row格式保存binlog；mysql主要依据sql语句来判断日志的保存格式，遇到表结构变更的时候就会以statement模式来记录。至于update或者delete等修改数据的语句，还是会记录所有行的变更。
